# 作業手順と検証チェック

## 編集前

1. 変更したい挙動を、どのシーン・スクリプトが所有しているか確認する。
2. `$NodePath` を変更する前に、対応する `.tscn` のノード名を確認する。
3. 現在の構成に合う最小変更を優先する。
4. 生成物である `.godot/` は編集しない。
5. 変更対象が `.tscn` を含む場合、参照切れを起こさないか確認する。

## スクリプトを変更するとき

- メソッドシグネチャには型を付ける。
- すべてのシーン接続も更新する場合を除き、signal 名は維持する。
- シーン側も更新する場合を除き、エクスポート変数は維持する。
- 入力挙動を変更する場合、新しい入力アクションが本当に必要なときだけ `project.godot` の Input Map を更新する。
- タイマー、オーディオ、UI については、重複ノードを作るより既存ノードの利用を優先する。
- リファクタリングとバランス変更を同時に行わない。

## シーンを変更するとき

- `.tscn` の差分は最小限にする。
- 可能な限り、`uid://...` 参照と既存リソースパスを維持する。
- フォーマット目的だけで大きなシーンセクションを並べ替えない。
- ノード名を変更したら、すべてのスクリプト内 `$NodePath` と、そのノードを使う signal 接続を更新する。
- ノードを削除したら、対応する `ext_resource` が未使用になっていないか確認する。

## 今回のプロジェクト固有チェック

- 戦闘画面に `PassiveFlowerSpawner` を復活させていない。
- status message の動的表示機能を追加していない。
- タイトルの未実装 UI を勝手に完成させていない。
- `selected_stage_id` または `StageDefinition` の戦闘接続を壊していない。
- `skill_id` の直書きを増やしていない。
- `RunState` や `BattleStartContext` を導入する場合、段階的に移行している。

## 推奨検証

ローカル環境で実行できる範囲のうち、最も強い確認を行う。

```bash
godot --headless --path . --quit
```

実行ファイル名が異なる環境では、インストール済み Godot の名前も試す。

```bash
godot4 --headless --path . --quit
```

Web エクスポートを変更した場合は、エクスポートテンプレートが入っているときだけ検証する。

```bash
godot --headless --path . --export-release Web ../Build/index.html
```

Godot CLI やエクスポートテンプレートが使えない場合は、検証できなかったことを報告する。  
コマンドを実行していないのに成功したと主張しない。
