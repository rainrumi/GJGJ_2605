# 作業手順と検証チェック

## 編集前

1. 変更したい挙動を、どのシーン・スクリプトが所有しているか確認する。
2. `$NodePath` を変更する前に、対応する `.tscn` のノード名を確認する。
3. 現在の構成に合う最小変更を優先する。
4. 生成物である `.godot/` は編集しない。

## スクリプトを変更するとき

- メソッドシグネチャには型を付ける。
- すべてのシーン接続も更新する場合を除き、信号名は維持する。
- シーン側も更新する場合を除き、エクスポート変数は維持する。
- 入力挙動を変更する場合、新しい入力アクションが本当に必要なときだけ `project.godot` のInput Mapを更新する。
- タイマー、オーディオ、UIについては、重複ノードを作るより既存ノードの利用を優先する。

## シーンを変更するとき

- `.tscn` の差分は最小限にする。
- 可能な限り、`uid://...` 参照と既存リソースパスを維持する。
- フォーマット目的だけで大きなシーンセクションを並べ替えない。
- ノード名を変更したら、すべてのスクリプト内 `$NodePath` と、そのノードを使う信号接続を更新する。

## 推奨検証

ローカル環境で実行できる範囲のうち、最も強い確認を行う。

```bash
godot --headless --path . --quit
```

実行ファイル名が異なる環境では、インストール済みGodotの名前も試す。

```bash
godot4 --headless --path . --quit
```

Webエクスポートを変更した場合は、エクスポートテンプレートが入っているときだけ検証する。

```bash
godot --headless --path . --export-release Web ../Build/index.html
```

Godot CLIやエクスポートテンプレートが使えない場合は、検証できなかったことを報告する。コマンドを実行していないのに成功したと主張しない。

## レビュー用チェックリスト

最終報告前に確認すること。

- ゲームは引き続き `main.tscn` から開始する。
- `HUD.start_game` は引き続き `Main.new_game()` に届く。
- `Player.hit` は引き続き `Main.game_over()` に届く。
- タイマーは意図された順序で開始・停止する。
- 新しいゲームプレイコードは、既存のスクリプト責務分担を守っている。
- 生成物である `.godot/` は変更していない。
